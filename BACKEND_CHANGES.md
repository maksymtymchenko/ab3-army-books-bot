# Зміни в бекенді для бота (замовлення / бронювання)

Щоб бот міг показувати **кому видана книга** і повну інфу по підтверджених замовленнях, потрібно доповнити відповіді API полями, які ти вже зберігаєш при створенні бронювання.

---

## 1. Що змінити

### 1.1. GET /api/reservations (список бронювань)

**Зараз** кожен елемент у `items` має лише:
- `id`, `bookId`, `status`, `createdAt`

**Потрібно додати до кожного елемента:**

| Поле       | Тип    | Опис                                      |
|-----------|--------|-------------------------------------------|
| `fullName`| string | ПІБ того, хто замовив (як у POST body)    |
| `phone`   | string | Телефон                                   |
| `subdivision` | string | Підрозділ (наприклад «1 ОМБр»)        |
| `comment` | string | (опційно) Коментар до замовлення          |

**Приклад оновленої відповіді `200`:**

```json
{
  "items": [
    {
      "id": "res_123",
      "bookId": "699e0662accb502370a2bb47",
      "status": "confirmed",
      "createdAt": "2026-02-23T10:15:00.000Z",
      "fullName": "Псевдо Боєць",
      "phone": "+380123456789",
      "subdivision": "1 ОМБр",
      "comment": "Буду в бібліотеці у пʼятницю."
    }
  ],
  "page": 1,
  "pageSize": 20,
  "totalItems": 1,
  "totalPages": 1
}
```

Якщо для старих записів цих полів немає в БД — можна повертати `null` або порожній рядок, бот це коректно покаже.

---

### 1.2. GET /api/reservations/{id} (одне бронювання)

**Зараз** тіло відповіді: `id`, `bookId`, `status`, `createdAt`.

**Потрібно додати ті самі поля:**  
`fullName`, `phone`, `subdivision`, `comment` (опційно).

**Приклад оновленої відповіді `200`:**

```json
{
  "id": "res_123",
  "bookId": "699e0662accb502370a2bb47",
  "status": "confirmed",
  "createdAt": "2026-02-23T10:15:00.000Z",
  "fullName": "Псевдо Боєць",
  "phone": "+380123456789",
  "subdivision": "1 ОМБр",
  "comment": "Буду в бібліотеці у пʼятницю."
}
```

---

## 2. Опційно (для меншого числа запитів з бота)

Якщо в відповіді **GET /api/reservations** (і при бажанні **GET /api/reservations/{id}**) додати вкладене об’єкт **книги**, бот зможе не викликати окремо `GET /api/books/{id}` для кожної книги.

До кожного елемента в `items` (або до одного об’єкта в GET by id) додати, наприклад:

```json
"book": {
  "id": "699e0662accb502370a2bb47",
  "title": "В сталевих грозах",
  "author": "Ернст Юнгер",
  "status": "reserved"
}
```

Це не обов’язково: зараз бот уже отримує книгу через `GET /api/books/{id}`, але з `book` у відповіді запитів буде менше.

---

## 3. Підсумок

| Що змінити | Дія |
|------------|-----|
| **GET /api/reservations** | У кожному елементі `items` повертати `fullName`, `phone`, `subdivision`, за бажанням `comment`. |
| **GET /api/reservations/:id** | У тілі відповіді повертати ті самі поля. |
| **(опційно)** Обидва ендпоінти | Додати вкладений об’єкт `book` з `id`, `title`, `author`, `status`. |

Після цих змін у бекенді можна оновити типи та логіку в боті, щоб замість «Кому видана: —» показувати, наприклад: **ПІБ, телефон, підрозділ**.

---

## 4. Статус «Повернено» (returned)

У боті під кожним підтвердженим замовленням є кнопка **«📥 Повернено»**. Вона надсилає:

- **PATCH /api/reservations/:id/status**  
  Body: `{ "status": "returned" }`

Щоб це працювало, у бекенді потрібно:

- Додати **`returned`** до допустимих значень статусу бронювання (наприклад у enum/константі `RESERVATION_STATUS` і в валідації Joi).
- При зміні на `returned` оновлювати статус бронювання і при потребі повертати книгу в статус `in_stock` (за твоєю бізнес-логікою).

---

## 5. Сповіщення в боті про нове бронювання

Бот піднімає HTTP-сервер (порт за замовчуванням **3131**), щоб приймати webhook від бекенду. Коли створюється нове бронювання, бекенд надсилає **POST** — тоді повідомлення приходить **усім користувачам бота** (усім, хто колись писав боту; їхні chat ID зберігаються в `data/chat-ids.json`).

### Що зробити в боті (вже реалізовано)

- Конфігурація **ADMIN_CHAT_ID** не потрібна.
- У **.env** за бажанням:
  - **`NOTIFY_WEBHOOK_SECRET`** — секрет для захисту ендпоінту; якщо заданий, бекенд повинен передавати його в заголовку або в body.
  - **`NOTIFY_PORT`** — порт сервера сповіщень, за замовчуванням `3131`.
- Після запуску бота сервер сповіщень слухає на `http://<host>:3131/notify/new-reservation`.

### Що зробити в бекенді

Після успішного створення бронювання (відповідь `201` з `POST /api/reservations`) викликати:

- **URL:** `POST http://<хост бота>:<NOTIFY_PORT>/notify/new-reservation`  
  Наприклад: `http://localhost:3131/notify/new-reservation` або `https://your-bot-host.com/notify/new-reservation` (якщо бот за деплоєм з відкритим портом/проксі).

- **Headers:** `Content-Type: application/json`  
  Якщо в боті заданий `NOTIFY_WEBHOOK_SECRET`, додати заголовок:  
  `X-Notify-Secret: <NOTIFY_WEBHOOK_SECRET>`  
  (або передати в body поле `secret`).

- **Body (JSON):**

```json
{
  "id": "res_123",
  "bookId": "699e0662accb502370a2bb47",
  "fullName": "Псевдо Боєць",
  "phone": "+380123456789",
  "subdivision": "1 ОМБр",
  "comment": "Буду в бібліотеці у пʼятницю.",
  "createdAt": "2026-02-23T10:15:00.000Z",
  "book": {
    "title": "В сталевих грозах",
    "author": "Ернст Юнгер"
  }
}
```

Обов’язкові поля: **`id`**, **`bookId`**, **`createdAt`**. Решта — опційно (у боті будуть показані як «—», якщо немає).

У відповідь бот повертає `200 { "ok": true, "sent": N }` (N — кількість чатів, куди відправлено) при успіху або помилку (400/401/500).
